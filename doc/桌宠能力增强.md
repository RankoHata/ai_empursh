# 桌面助手 2D 角色动态系统迁移方案需求文档

> **项目名称**：ai_empursh 桌面助手 - 角色渲染引擎迁移  
> **版本**：v1.0  
> **状态**：规划中  
> **目标**：将现有 Live2D 驱动的 G36 角色替换为基于 Spine 的高动态 2D 立绘（仿 NIKKE 风格），实现更丰富的动作交互与性能优化。

---

## 1. 项目背景与目标

### 1.1 现状
- 当前桌面助手使用 **Live2D Cubism SDK（Web）** 加载并展示 G36 角色的动态立绘。
- 存在以下痛点：
  - 官方 SDK 兼容性差，需频繁修改源码。
  - 动画表现力有限（幅度小、表情细腻但动作僵硬）。
  - 无法实现类似 NIKKE 的大幅度战斗/待机动作。
  - 性能开销较高，在低配设备上卡顿。

### 1.2 目标
- **技术升级**：彻底移除 Live2D 相关依赖，改用 **Spine（骨骼动画）** 作为角色渲染核心。
- **体验提升**：实现仿 NIKKE 风格的动态立绘，支持：
  - 流畅的待机、点击、拖拽、视线跟随等交互。
  - 多动作无缝切换（如挥手、转枪、打招呼）。
  - 更丰富的物理模拟（头发、衣物摆动）。
- **可维护性**：采用社区成熟的 `PixiJS + pixi-spine` 方案，降低底层维护成本。

---

## 2. 现有代码分析（基于仓库结构推测）

- **Live2D 相关文件/目录**（需移除）：
  - `src/live2d/` 或类似目录，包含模型加载、渲染循环、交互逻辑。
  - `public/models/G36/` 存放 `.moc3`、`.model3.json`、贴图等。
  - 依赖项：`live2dcubismcore.min.js`、`pixi-live2d-display`（如果使用）等。
- **交互接口**：
  - 当前可能存在 `onClick`、`onMouseMove` 等事件绑定，用于触发动画或表情。
- **主入口**：
  - `main.js` 或 `App.vue` 中初始化 Live2D 渲染器并挂载到 DOM。

---

## 3. 技术方案概述

### 3.1 核心技术栈
| 组件 | 选型 | 说明 |
|------|------|------|
| 2D渲染引擎 | PixiJS v7 / v8 | 轻量高效，WebGL加速 |
| Spine渲染插件 | pixi-spine（官方维护） | 与Pixi深度集成，支持Spine 3.8+ |
| 动画状态机 | 自定义或使用Spine状态机 | 管理动作切换与混合 |
| 交互检测 | 基于Spine边界框（Bounding Box） | 精准点击/悬停区域 |
| 资源加载 | PixiJS Assets | 统一管理模型、图集、音效等 |

### 3.2 模型资源
- 目标模型：NIKKE 风格角色立绘（Spine 格式）。
- 文件格式：`.skel`（二进制）或 `.json` + `.atlas` + `.png`。
- **获取途径**：GitHub 社区已有提取资源（需自行研究，本方案仅规划技术集成，不涉及版权指导）。

### 3.3 交互能力设计
- **点击交互**：点击角色不同部位（头、身体、手）触发预设动作。
- **鼠标悬浮**：改变表情或播放轻量动画。
- **视线跟随**：角色眼睛跟随鼠标位置移动（通过修改骨骼旋转）。
- **拖拽**：可拖动角色在桌面任意位置移动（与窗口拖拽逻辑结合）。

## 4. 迁移实施步骤

### 4.1 阶段一：移除 Live2D 遗留代码（清理）
- 删除所有 Live2D 相关依赖包（`package.json` 中移除）。
- 移除 Live2D 模型资源文件（`public/models/G36/` 等）。
- 删除 Live2D 渲染、加载、交互的源码文件。
- 移除 Live2D 配置项（如环境变量中的模型路径）。

### 4.2 阶段二：集成 Spine 渲染环境

#### 4.2.1 安装依赖
```bash
npm install pixi.js pixi-spine
```

#### 4.2.2 初始化 PixiJS 应用（替代原 Live2D 渲染器）
- 创建 `PIXI.Application` 实例，设置透明背景（若桌面助手需透明）。
- 挂载 canvas 到指定 DOM 容器。

#### 4.2.3 加载 Spine 模型
- 使用 `PIXI.Assets.load` 加载 `.skel` 或 `.json` 文件。
- 实例化 `new PIXI.spine.Spine(spineData)`。
- 添加到舞台，设置初始位置/缩放。

### 4.3 阶段三：实现交互逻辑
- 启用 `interactive` 属性。
- 监听 `click`、`pointermove` 事件。
- 在事件回调中调用 `hitTest(点坐标)` 获取命中的骨骼/边界框名称。
- 根据名称执行不同的动作切换（`spine.state.setAnimation`）。

### 4.4 阶段四：对接原有桌面助手功能
- 保持语音、聊天、定时提醒等原有逻辑不变，仅替换角色显示模块。
- 若原有功能需要与角色互动（如播报时口型同步），需重新实现口型驱动（可基于音视频分析或预定义动画）。

### 4.5 阶段五：性能优化与兼容性测试
- 启用 PixiJS 的 `autoDensity` 适配屏幕分辨率。
- 限制帧率（如 60fps）。
- 测试 Windows/macOS 下 Electron 或 Web 环境的稳定性。

## 7. 风险与注意事项

- **模型兼容性**：确保获取的 Spine 模型版本与 `pixi-spine` 插件版本匹配（建议 3.8+）。
- **资源获取**：若从网上下载 NIKKE 模型，需注意文件完整性（图集、贴图、骨骼数据需对应）。
- **交互区域定义**：若模型未预置边界框，需在 Spine 编辑器中自定义或通过坐标近似判断。
- **内存泄漏**：切换模型或销毁时需正确释放资源（`destroy()`）。
- **原有功能回归**：迁移后需全面回归测试桌面助手的其他功能。

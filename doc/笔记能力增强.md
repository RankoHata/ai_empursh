# AI 桌面助理知识库能力增强 - 需求与概要设计说明书

**项目名称**：AI Empursh - 知识库模块重构
**版本**：V2.0
**日期**：2026-06-21
**核心原则**：物理隔离、安全优先、本地索引、AI 透明（零接触）

---

## 1. 引言与背景

### 1.1 项目现状
当前桌面助理（`ai_empursh`）的“笔记”功能实质为“聊天对话摘录”，仅支持从聊天消息右键保存，缺乏独立的文本收藏、外部文件挂载及隐私隔离能力。

### 1.2 项目目标
将现有笔记模块升级为 **“双轨制（普通/秘密）个人知识库助理”**，实现以下核心能力：
1. **通用文本收藏**：手动保存任意文本，支持标签与全文检索。
2. **高隐私数据隔离**：敏感文本独立存储，绝对禁止流向外部大模型（LLM）。
3. **外部工作区挂载**：支持挂载本地目录（含 Obsidian 仓库），自动扫描并索引 Markdown 文件，按安全级别分流存储。

---

## 2. 总体需求概述

经过需求分析与整合（原需求3“Obsidian联动”与需求4“AI工作目录”在底层均为“本地目录扫描”，故合并），最终形成 **3 大核心需求 + 1 个基础设施底座**：

| 编号 | 需求名称 | 核心描述 | 数据流向 |
| :--- | :--- | :--- | :--- |
| **需求1** | **通用文本收藏与检索** | 手动添加任意文本，打标签。支持自然语言检索（走 LLM）。 | 存入 `data.db`（普通库），LLM 可见上下文。 |
| **需求2** | **高隐私文本隔离存储** | 通过专用 UI 保存/搜索高敏感文本。**绝对不走外部 LLM**。 | 存入 `secret.db`（秘密库），LLM 仅见占位符。 |
| **需求3** | **外部工作区挂载与索引** | 配置本地目录，自动扫描 `.md` 文件，根据 `scope` 自动分流索引。 | 分流至 `data.db` 或 `secret.db`。 |
| **底座** | **双库引擎 + 解析器** | 物理隔离的双 SQLite 数据库 + FTS5 全文索引 + Markdown 解析剥离。 | 底层支撑。 |

## 3. 详细需求说明

### 3.1 需求1：通用文本收藏与检索（增强）
- **输入**：用户在前端“新建笔记”弹窗中输入文本内容及标签（逗号分隔）。
- **处理**：后端直接接收 `type: 'add_note'` 请求，调用工具函数写入 `data.db`。**此过程不经 LLM**。
- **检索**：用户可在聊天框输入“帮我找一下关于 Python 的笔记”。LLM 通过 Function Calling 提取关键词 `Python`，调用 `search_notes` 工具，将结果返回给用户（普通数据可正常展示在聊天上下文中）。

### 3.2 需求2：高隐私文本隔离存储（秘密空间）
- **交互隔离**：前端必须提供独立的 **“秘密笔记面板”**（带 🔒 标识）。该面板拥有独立的新建输入框和搜索框。
- **写入红线**：用户在秘密面板输入的内容（如 `我的银行卡号是 1234`），通过 `type: 'add_secret_note'` 直接发送至后端。后端直接操作 `secret.db`，**绝不写入 LLM 的 `messages` 历史数组**。
- **查询红线**：若用户在聊天框（而非专用搜索框）试图查询秘密数据，后端执行查询后将 **真实数据直推前端 UI 渲染**，但只向 LLM 返回脱敏占位符（如 `"已检索到 3 条秘密记录，详情请查看安全面板"`）。

### 3.3 需求3：外部工作区挂载与索引（原需求3+4合并）
- **配置方式**：在 `config.yaml` 中注册目录，并标记 `scope`（安全级别）。
- **扫描范围**：当前仅支持 `.md` 文件。解析时需剥离 YAML Front Matter 和 Markdown 语法（如 `#`、`**`、`-` 列表标记），仅提取纯文本用于 FTS 索引，原始 Markdown 保留用于导出。
- **同步机制**：
  - **全量扫描**：服务启动时执行。
  - **增量同步**：后台定时（如每 5 分钟）或用户手动触发“刷新索引”。
  - **变更检测**：基于文件系统 `mtime`（修改时间）与 `file_hash`（SHA-256）组合判断。仅更新有变动的文件，避免全量重建性能开销。
  - **删除清理**：扫描时发现文件物理消失，自动将数据库记录标记为删除或级联清理 FTS。

### 4.2 物理存储层（双库隔离）
- **普通库 (`data.db`)**：存放需求1手动笔记 + `scope: public` 的外部文件索引。
- **秘密库 (`secret.db`)**：存放需求2手动秘密笔记 + `scope: secret` 的外部文件索引。
- **结构一致性**：两库表结构完全相同，代码层复用数据模型，仅连接字符串不同。

### 4.3 核心安全机制：数据流与路由
这是防止秘密数据泄露给外部 LLM 的关键：

1. **写入阶段**：
   - 秘密面板的 `add_secret_note` 请求在网关层被直接截获，调用 `secret_notes.add_note()` 后立即 `return`，不进入 `process_chat` 流程。
2. **查询阶段（聊天场景）**：
   - 用户问：“帮我查秘密里的游戏资料”。
   - LLM 调用 `search_secret_notes` 工具。
   - 后端执行 `secret.db` 查询，得到真实结果 `R`。
   - **分叉处理**：
     - `ws.send(real_data_R)` → 推送给前端面板展示。
     - `return placeholder("已找到 N 条结果，已推送至安全面板")` → 仅此字符串拼接到 LLM 的上下文历史中。
3. **绝对阻断**：整个 Python 后端进程中，秘密数据字符串**绝不**出现在构造 HTTP 请求体（调用 OpenAI API 时）的代码路径上。

### 4.4 外部同步服务（Scanner Service）
- **触发时机**：启动时自动运行，并暴露 `manual_sync` WebSocket 接口供前端“刷新”按钮调用。
- **处理流程**：
  1. 遍历 `config.yaml` 中 `workspaces` 列表。
  2. 对每个 `.md` 文件，读取 `mtime` 并与数据库 `file_hash` 比对。
  3. 若有变动，调用 `MarkdownParser` 提取纯文本与元数据。
  4. 根据目录的 `scope` 属性，调用对应的数据库工具写入 `data.db` 或 `secret.db`。
  5. 更新或插入 `knowledge_items` 记录，并自动同步 FTS5 虚拟表（通过 SQLite 触发器或手动 `INSERT`）。

## 5. 数据模型设计（表结构）

### 5.1 核心知识条目表 (`knowledge_items`)
*该表同时存在于 `data.db` 和 `secret.db` 中，结构一致。*

```sql
CREATE TABLE knowledge_items (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_type TEXT NOT NULL CHECK(source_type IN ('manual', 'external_file')),
    source_path TEXT,                           -- 外部文件绝对路径，手动记录为 NULL
    title TEXT,                                 -- 文件标题或手动输入的标题
    content_raw TEXT NOT NULL,                  -- 原始 Markdown 或手动文本（保留格式）
    content_plain TEXT NOT NULL,                -- 剥离语法后的纯文本（用于 FTS 及预览）
    tags TEXT,                                  -- 标签存储（JSON 数组或逗号分隔）
    file_mtime INTEGER,                         -- 外部文件修改时间戳（用于增量同步）
    file_hash TEXT,                             -- 文件 SHA-256（防重校验）
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 5.2 FTS5 全文索引虚拟表
```sql
CREATE VIRTUAL TABLE items_fts USING fts5(
    title,
    content_plain,
    content=knowledge_items,                    -- 关联外部内容表
    content_rowid=id
);

-- 触发器（自动同步新增/更新/删除）
CREATE TRIGGER items_ai AFTER INSERT ON knowledge_items BEGIN
    INSERT INTO items_fts(rowid, title, content_plain) VALUES (new.id, new.title, new.content_plain);
END;
-- （同理需建 AFTER UPDATE 和 AFTER DELETE 触发器）
```

### 5.3 标签独立表（用于标签统计与快速筛选）
```sql
CREATE TABLE tags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL
);
CREATE TABLE note_tag (
    note_id INTEGER,
    tag_id INTEGER,
    FOREIGN KEY (note_id) REFERENCES knowledge_items(id) ON DELETE CASCADE,
    FOREIGN KEY (tag_id) REFERENCES tags(id) ON DELETE CASCADE,
    PRIMARY KEY (note_id, tag_id)
);
```
*注：为简化 V1.0 实现，`knowledge_items` 中的 `tags` 字段可先使用纯文本逗号分隔，暂不强制三范式，视后续性能需求再拆表。*

## 7. 安全与权限设计总结

### 7.1 配置示例 (`config.yaml`)
```yaml
workspaces:
  - path: "/Users/me/Documents/Obsidian_Vault"
    scope: "public"
    enabled: true
    sync_interval: 300  # 秒
  - path: "/Users/me/Work/Secret_Contracts"
    scope: "secret"
    enabled: true
    sync_interval: 60

llm:
  external_provider: "openai"
  internal_provider: "ollama"  # 预留
  allow_secret_to_internal: false  # 即使内部模型，默认也禁止，需手动开启
```

### 7.2 秘密数据“零接触”保障清单
1. ✅ **物理隔离**：秘密数据存入独立 `.db` 文件，操作系统的文件权限即可区分。
2. ✅ **管道隔离**：WebSocket 网关根据 `type` 前缀硬路由，秘密操作不经过 LLM 流程。
3. ✅ **上下文净化**：即便 LLM 调用了秘密工具，返回给 LLM 的 `messages` 中也仅有脱敏占位符。
4. ✅ **前端警示**：秘密面板 UI 有显著红色标识，提醒用户当前操作不经过大模型。

## 8. 未来扩展预留

- **解析器插件化**：当前仅实现 `MarkdownParser`，未来可通过注册 `PDFParser`（依赖 `pypdf`）和 `DocxParser`（依赖 `python-docx`），无需修改同步调度代码。
- **向量化检索**：在 `knowledge_items` 表中预留 `embedding` 字段（BLOB），配合内部本地模型实现 RAG（检索增强生成）。
- **全文检索增强**：当前 FTS5 基于分词，未来若需中文语义搜索，可接入 `jieba` 分词器或升级为 `sqlite-vec` 插件。

## 9. 附录：开发实施建议（迭代顺序）

建议按以下顺序进行开发，以降低风险并快速验证：

1. **Phase 1**：拆分双数据库连接工厂，将现有 `notes.py` 改造为 `public_notes.py` 并强制传入 `scope`。
2. **Phase 2**：实现需求1前端“新建笔记”按钮及 `add_note` 直连逻辑。
3. **Phase 3**：实现 `secret_notes.py` + 前端独立秘密面板（UI 硬隔离）。
4. **Phase 4**：实现 `MarkdownParser` 和 `workspace_sync.py`，完成外部目录挂载与分流。
5. **Phase 5**：实现 LLM 上下文占位符净化逻辑（安全围栏）。

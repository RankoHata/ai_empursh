## 3. 核心功能设计详述

### 3.1 功能1：人格多版本管理（按需配置）

#### 3.1.1 设计理念

系统支持为同一个角色（如“阿尼斯”）配置多个版本（Simple / Full / Arc），但**不强制要求配齐所有版本**。用户可按需配置 1 个、2 个或 3 个版本。

| 配置场景 | 示例 | 前端表现 |
|----------|------|----------|
| 只配 1 个版本 | 阿尼斯（Simple） | 人格列表中只显示“阿尼斯”，无额外选项 |
| 配 2 个版本 | 阿尼斯（Simple + Full） | 人格名称旁显示版本切换下拉菜单 |
| 配 3 个版本 | 阿尼斯（Simple + Full + Arc） | 人格名称旁显示版本切换下拉菜单 |

#### 3.1.2 数据模型扩展

在现有 `personalities` 表中增加以下字段：

```sql
ALTER TABLE personalities ADD COLUMN parent_id INTEGER;      -- 关联同一角色的不同版本，NULL 表示独立人格
ALTER TABLE personalities ADD COLUMN version_tag TEXT;       -- 'simple' | 'full' | 'arc' | NULL
ALTER TABLE personalities ADD COLUMN metadata JSON;          -- 扩展信息（情绪映射、模板变量默认值等）
```

#### 3.1.3 配置示例

| 角色名 | parent_id | version_tag | 前端显示 |
|--------|-----------|-------------|----------|
| 默认助手 | NULL | NULL | “默认助手” |
| 阿尼斯 | NULL（或自关联） | 'simple' | “阿尼斯” |
| 拉毗 | 101 | 'full' | “拉毗 (完整)” |
| 拉毗 | 101 | 'arc' | “拉毗 (深度)” |
| 尼恩 | 102 | 'simple' | “尼恩 (简洁)” |
| 尼恩 | 102 | 'full' | “尼恩 (完整)” |
| 尼恩 | 102 | 'arc' | “尼恩 (深度)” |

#### 3.1.4 API 行为

- **加载人格**：`GET /personalities/{id}` 返回单个版本详情
- **加载角色所有版本**：`GET /personalities?parent_id={id}` 返回同一角色的所有版本
- **前端聚合**：前端根据 `parent_id` 聚合，自动决定展示单版本还是多版本下拉

### 3.2 功能3：System Prompt 模板引擎（Jinja2）

#### 3.2.1 设计目标

让 System Prompt 支持动态变量，增强人格的个性化表达能力。

#### 3.2.2 实现方案

- 引入 `jinja2` 库
- 在 `agent/chat.py` 加载 System Prompt 后，调用模板渲染
- 支持的默认变量：
  - `{{ user_name }}`：用户名称（从前端传入或从数据库读取）
  - `{{ current_time }}`：当前时间
  - `{{ personality_name }}`：当前人格名称
  - `{{ version_tag }}`：当前版本标签（simple/full/arc）

#### 3.2.3 使用示例

```yaml
system_prompt: |
  你是 {{ personality_name }}，一个 {{ version_tag }} 版本的 AI 助理。
  现在是 {{ current_time }}，请用亲切的语气称呼用户为 {{ user_name }}。
```

#### 3.2.4 代码改动点

```python
# backend/utils/template.py (新增)
from jinja2 import Template

def render_prompt(prompt: str, context: dict) -> str:
    template = Template(prompt)
    return template.render(context)
```

### 3.3 功能4：上下文动态路由（自动选版本）

#### 3.3.1 设计目标

当角色配置了多个版本时，系统根据当前对话内容**自动选择**最合适的版本，无需用户手动切换。

#### 3.3.2 路由逻辑

在 `agent/chat.py` 中增加路由函数，输入用户消息和对话历史，输出推荐的 `version_tag`：

```python
def route_version(user_input: str, history: list, available_versions: list) -> str:
    # 如果只有一个版本，直接返回
    if len(available_versions) == 1:
        return available_versions[0]
    
    # 简单关键词匹配
    deep_keywords = ["剧情", "回忆", "过去", "为什么", "曾经", "内心", "真正"]
    full_keywords = ["详细", "告诉", "怎样", "如何", "觉得"]
    
    input_lower = user_input.lower()
    
    # 优先级：Arc > Full > Simple
    if any(kw in input_lower for kw in deep_keywords) and 'arc' in available_versions:
        return 'arc'
    elif any(kw in input_lower for kw in full_keywords) and 'full' in available_versions:
        return 'full'
    else:
        # 默认返回最简版本，或 available_versions 的第一个
        return available_versions[0] if 'simple' not in available_versions else 'simple'
```

> **注**：后续可升级为基于 LLM 的判断，但一期采用轻量级关键词规则。

#### 3.3.3 前端配合

- 用户可在设置中开启/关闭“自动路由”功能（默认开启）
- 关闭时，固定使用用户手动选择的版本

### 3.4 功能7：Live2D 情绪状态同步（支持开关）

#### 3.4.1 设计目标

后端在生成回复时，附加情绪标签（`emotion`），通过 WebSocket 下发给前端，驱动 Live2D 模型切换表情/动作。前端必须提供**开关**控制此功能。

#### 3.4.2 情绪标签定义

一期支持以下情绪标签（与 Live2D 动作对应）：

| 情绪标签 | 说明 | Live2D 动作建议 |
|----------|------|----------------|
| `idle` | 默认待机 | 待机动画 |
| `happy` | 开心/高兴 | 微笑、挥手 |
| `sad` | 悲伤/低落 | 低头、垂眼 |
| `angry` | 生气/不满 | 皱眉、叉腰 |
| `thinking` | 思考/认真 | 托腮、歪头 |
| `surprised` | 惊讶 | 睁大眼、后退 |
| `bored` | 无聊/嫌弃 | 翻白眼、叹气 |

#### 3.4.3 后端情绪生成逻辑

在 `agent/chat.py` 中，根据对话内容或用户输入，生成情绪标签：

```python
def detect_emotion(user_input: str, assistant_response: str) -> str:
    # 简单规则匹配
    happy_keywords = ["开心", "高兴", "喜欢", "谢谢", "棒"]
    sad_keywords = ["难过", "伤心", "失望", "遗憾"]
    angry_keywords = ["生气", "讨厌", "烦", "滚"]
    think_keywords = ["为什么", "怎么", "思考", "觉得"]
    
    if any(kw in assistant_response for kw in happy_keywords):
        return "happy"
    elif any(kw in assistant_response for kw in sad_keywords):
        return "sad"
    elif any(kw in assistant_response for kw in angry_keywords):
        return "angry"
    elif any(kw in assistant_response for kw in think_keywords):
        return "thinking"
    else:
        return "idle"
```

> **注**：后期可升级为调用情感分析小模型，一期采用轻量级关键词规则。

#### 3.4.4 WebSocket 协议扩展

在 WebSocket 消息中增加 `emotion` 字段：

```json
{
    "type": "response",
    "content": "你好呀主人！今天心情不错呢。",
    "emotion": "happy",
    "done": false
}
```

流式响应中，仅在 `done: true` 时下发最终情绪标签。

#### 3.4.5 前端开关设计

在设置面板增加开关：**“Live2D 情绪跟随”**

| 开关状态 | 行为 |
|----------|------|
| **开启（默认）** | 前端监听 WebSocket 的 `emotion` 字段，切换 Live2D 动作 |
| **关闭** | 前端忽略 `emotion` 字段，Live2D 保持默认 `idle` 状态 |

> 即使开关关闭，后端仍然生成情绪标签，保证用户重新打开开关时无需重新加载对话即可生效。
